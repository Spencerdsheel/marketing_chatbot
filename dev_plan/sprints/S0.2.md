## Goal
Stand up the runnable FastAPI application that every later module plugs routers into. It loads
`services/common` config (fail-fast), emits JSON logs, generates/propagates a correlation ID, maps
`AppException` to the standard error envelope, and exposes `/healthz`, `/readyz` (DB + Redis), `/metrics`.
This is the **first Postman-testable sprint**.

## Decisions (locked)
- New package **`services/api`**, `src/` layout, import package name **`api`**, distribution `chatbot-api`.
  Mirror `services/common/pyproject.toml` for ruff (`E,F,I,UP,B,S,ASYNC`, line-length 100) + mypy
  (`strict = true`) + pytest (`asyncio_mode=auto`). Depends on `chatbot-common`, `fastapi`,
  `uvicorn[standard]`. Dev deps mirror common (`pytest`, `pytest-asyncio`, `ruff`, `mypy`, plus `httpx` for
  the ASGI test client).
- **App factory** `create_app() -> FastAPI` in `services/api/src/api/app.py`. Run via
  `uvicorn api.app:create_app --factory`.
- **Config:** reuse `common.settings.get_settings()` (no new Settings class). Set the service name via the
  `SERVICE_NAME` env var = `api`.
- **DB through PgBouncer:** connect with `statement_cache_size=0` (required for asyncpg + PgBouncer
  transaction mode). This needs a small foundation change (below).
- **/readyz** returns HTTP **200** when all checks pass, **503** when any fail (body always lists per-check
  status). Never fail silently.

## Foundation changes (services/common — small, tested, justified)

2. `common/health.py` — add `async def check_redis(client) -> bool` mirroring `check_database`: returns

   (fake client with ping → True/raise).

## Scope (build) — services/api
1. `create_app()` wiring, in this order:
   - `get_settings()` at startup (fail-fast on missing/invalid env).
   - Configure root/app loggers via `common.logging.get_logger`.
   - **Lifespan**: on startup connect the database via `common.db.Database.connect(<dsn from settings>,
     statement_cache_size=0)` and, when a Redis URL is configured, create a `redis.asyncio` client for health;
     store both on `app.state`. On shutdown close both. (Cache wiring via `build_cache` may be stored too but
     isn't required for this sprint.)
   - **Correlation-ID middleware**: read `X-Correlation-Id` request header or generate `uuid4().hex`; bind it
     with `common.logging.log_context(correlation_id=...)` for the request; set it on
     `response.headers["X-Correlation-Id"]`.
   - **Error handling**: an exception handler for `AppException` → JSON `{error_code, message,
     correlation_id}` with `exc.http_status` (include `Retry-After` for `RateLimitError` when set); a
     catch-all handler for unhandled exceptions → 500 envelope built from `InternalServerError`, logging full
     detail server-side (never leak internals to the client).
2. Routes:
   - `GET /healthz` → `common.health.liveness()` (200, no dependency checks).
   - `GET /readyz` → build checks `{"database": lambda: check_database(app.state.db)}` plus
     `{"redis": lambda: check_redis(app.state.redis)}` when redis is configured; call
     `common.health.readiness(checks)`; return body `{"ready": bool, "checks": {...}}` with status 200/503.
   - `GET /metrics` → `common.health.metrics_payload()` returned with its content-type.

## Tests (TDD, services/api/tests/unit) — write first
Use `httpx.AsyncClient(transport=ASGITransport(app=...))`. Design seams so checks read from `app.state` and
tests inject fakes (a fake db whose `fetchval` returns 1 / raises; a fake redis whose `ping` returns True /
raises).
- `/healthz` → 200 `{"status":"ok"}`.
- `/readyz` ready → 200, `checks.database == "ok"` (and redis ok when configured).
- `/readyz` not-ready → 503 when the db check raises/returns false.
- `/metrics` → 200 with Prometheus content-type.
- Correlation ID: response always carries `X-Correlation-Id`; when the request supplies one, it is echoed back.
- Error envelope: a temporary test-only route raising `NotFoundError` → 404 `{error_code:"NOT_FOUND", message,
  correlation_id}`; a route raising a bare `Exception` → 500 `{error_code:"INTERNAL_ERROR", ...}` and does not
  leak the original message.
- An optional `integration`-marked test (deselected unless a real DB/Redis are present) may hit live deps.

## Manual test recipe (user runs)
Pre-req: the S0.1 stack is up (postgres/pgbouncer/redis healthy).

1. **Config** — create a real `.env` at the repo root from `.env.example`:
   - `DATABASE_URL` → the PgBouncer DSN on port **6432** (the `.env.example` default already does this).
   - `REDIS_URL` → the redis URL on port 6379.
   - `DEPLOYMENT_MODE=saas`, `SERVICE_NAME=api`.
   - Replace the two secret **placeholders** with real values. Generate them with:
     - `python -c "import secrets; print(secrets.token_urlsafe(48))"` → paste as the JWT secret value.
     - `python -c "import secrets,base64; print(base64.urlsafe_b64encode(secrets.token_bytes(32)).rstrip(b'=').decode())"`
       → paste as the encryption-key value.
     (Placeholders won't pass validation — the encryption key must resolve to exactly 32 bytes.)
2. **Install** both packages editable into the project venv:
   - `venv\Scripts\python -m pip install -e services/common -e services/api`
3. **Run** the API:
   - `venv\Scripts\python -m uvicorn api.app:create_app --factory --port 8000`

Then in **Postman** (or browser/curl):
- `GET http://localhost:8000/healthz` → `200 {"status":"ok"}`.
- `GET http://localhost:8000/readyz` → `200 {"ready":true,"checks":{"database":"ok","redis":"ok"}}`.
- `GET http://localhost:8000/metrics` → `200`, Prometheus text exposition.
- `GET /healthz` with header `X-Correlation-Id: test-123` → response echoes `X-Correlation-Id: test-123`.
- (Negative) `docker stop chatbot-redis` → `GET /readyz` → `503` with `"redis":"fail"`; then restart it.
