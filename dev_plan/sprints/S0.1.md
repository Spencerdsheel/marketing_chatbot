## Goal
Stand up the local development backing services so every later backend sprint has a real Postgres (with
pgvector), Redis, and PgBouncer to run against. No application code yet тАФ this is pure infrastructure.

## Scope (build)
1. `deploy/docker-compose.dev.yml` with three services, all with healthchecks, sane dev defaults via
   `${VAR:-default}` so `docker compose up` works with zero config:
   - **postgres** тАФ image `pgvector/pgvector:pg16`. Env `POSTGRES_USER/PASSWORD/DB` (defaults
     `chatbot`/`chatbot`/`chatbot`). Named volume `pgdata`. Mount the init dir (below) into
     `/docker-entrypoint-initdb.d`. Port `${POSTGRES_PORT:-5432}:5432`. Healthcheck: `pg_isready`.
   - **pgbouncer** тАФ image `edoburu/pgbouncer`. **Transaction pool mode** (`POOL_MODE=transaction`).
     Points at the `postgres` service. Port `${PGBOUNCER_PORT:-6432}:6432`. `depends_on` postgres healthy.
     Healthcheck via a TCP/psql probe.
   - **redis** тАФ image `redis:7-alpine`. Port `${REDIS_PORT:-6379}:6379`. Healthcheck: `redis-cli ping`.
2. `deploy/postgres/init/01-extensions.sql` тАФ runs on first init:
   - `CREATE EXTENSION IF NOT EXISTS vector;` in the main DB.
   - Create `chatbot_test` database **and** enable `vector` in it (so `services/common` integration tests can
     use `TEST_DATABASE_URL`). Use a `.sh` wrapper if needed to create a second DB + run the extension in it.
3. Root **`.env.example`** тАФ a superset of `services/common/.env.example`, documenting:
   - The compose vars (`POSTGRES_USER/PASSWORD/DB`, `POSTGRES_PORT`, `PGBOUNCER_PORT`, `REDIS_PORT`).
   - `DATABASE_URL` for the **app via PgBouncer**: `postgresql://chatbot:chatbot@localhost:6432/chatbot`.
   - A documented **direct** DSN for **migrations + integration tests** (PgBouncer transaction mode can't run
     them): `postgresql://chatbot:chatbot@localhost:5432/chatbot` and the `chatbot_test` variant.
   - All `services/common` required vars (`DEPLOYMENT_MODE`, `REDIS_URL`, `REPOSITORY`, `JWT_SECRET`,
     `SECRET_ENCRYPTION_KEY`, `LOG_LEVEL`, `SERVICE_NAME`) with placeholder values тАФ never real secrets.
   - A comment noting asyncpg through PgBouncer transaction mode needs `statement_cache_size=0` (handled in
     S0.2 app wiring, just documented here).
4. `deploy/README.md` тАФ bring-up + verification commands (the test recipe below).

## Manual test recipe (user runs)
```bash
# from repo root
docker compose -f deploy/docker-compose.dev.yml up -d
docker compose -f deploy/docker-compose.dev.yml ps          # all three "healthy"

# pgvector present in main DB:
docker compose -f deploy/docker-compose.dev.yml exec postgres \
  psql -U chatbot -d chatbot -c "SELECT extversion FROM pg_extension WHERE extname='vector';"

# test DB exists with vector:
docker compose -f deploy/docker-compose.dev.yml exec postgres \
  psql -U chatbot -d chatbot_test -c "SELECT extversion FROM pg_extension WHERE extname='vector';"

# redis:
docker compose -f deploy/docker-compose.dev.yml exec redis redis-cli ping   # -> PONG

# pgbouncer reachable on 6432 (from host, if psql installed):
psql "postgresql://chatbot:chatbot@localhost:6432/chatbot" -c "select 1;"
```
Expected: `ps` shows 3 healthy services; both `vector` queries return a version (e.g. `0.7.x`); redis тЖТ `PONG`;
the pgbouncer query returns `1`.
