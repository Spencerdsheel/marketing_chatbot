## Goal
Establish the database migration tooling (Alembic) so every later module ships its tables as versioned
migrations. No business tables yet — this sprint only proves the harness: configured env, a baseline revision,
and a clean `upgrade head` / `downgrade base` round-trip against the real Postgres.

## Decisions (locked)
- **Location:** `services/api/migrations/` (Alembic env + `versions/`) with `services/api/alembic.ini`. The api
  package owns DB wiring, so migrations live with it. Run from the `services/api` directory (or via `-c`).
- **Driver:** Alembic runs **synchronously** with **psycopg2** (the app uses asyncpg; migrations use a sync
  driver — standard split). DSN scheme `postgresql://...` resolves to psycopg2 by default.
- **Target DB = DIRECT Postgres (port 5432), NOT PgBouncer.** PgBouncer transaction mode can't run DDL/DDL
  transactions reliably. `env.py` reads `DATABASE_URL_DIRECT` from the environment, falling back to
  `DATABASE_URL` only if the direct one is unset.
- **Raw-SQL migrations** (no ORM models / no autogenerate). Migrations use `op.execute(...)` /
  `op.create_table(...)` by hand. `env.py` therefore needs no model imports; `target_metadata = None`.
- **Baseline `0001_baseline` is a no-op** (empty `upgrade`/`downgrade`). It establishes the `alembic_version`
  table + the revision chain; real tables arrive in S1.1.

## Scope (build)
1. Add deps to `services/api/pyproject.toml`: `alembic`, `psycopg2-binary` (main deps — the api owns migrations).
2. `services/api/alembic.ini` — standard config; `script_location = migrations`; leave `sqlalchemy.url` empty
   (env.py sets it from the environment).
3. `services/api/migrations/env.py` — sync env that:
   - reads `DATABASE_URL_DIRECT` (fallback `DATABASE_URL`) from `os.environ`; raises a clear error if neither
     is set (fail-fast);
   - sets it as the SQLAlchemy URL; supports both offline and online modes; `target_metadata = None`.
4. `services/api/migrations/script.py.mako` — standard Alembic template.
5. `services/api/migrations/versions/0001_baseline.py` — `revision = "0001"`, `down_revision = None`,
   no-op `upgrade()`/`downgrade()`.
6. Document the commands in `deploy/README.md` (append a "Migrations" section) and in this sprint's test recipe.

## Manual test recipe (user runs)
Pre-req: S0.1 stack up; `.env` present (from S0.2) with `DATABASE_URL_DIRECT` = direct Postgres DSN on **5432**.
```powershell
# install the new migration deps into the venv
venv\Scripts\python -m pip install -e services/api

# run from the api package dir so alembic.ini + migrations/ are found
cd services\api
..\..\venv\Scripts\python -m alembic upgrade head        # -> creates alembic_version, stamps 0001
..\..\venv\Scripts\python -m alembic current             # -> shows 0001 (head)
..\..\venv\Scripts\python -m alembic downgrade base      # -> back to empty
..\..\venv\Scripts\python -m alembic upgrade head         # -> round-trips cleanly
cd ..\..
```
Confirm in the DB:
```powershell
docker compose -f deploy\docker-compose.dev.yml exec postgres psql -U chatbot -d chatbot -c "\dt"
```
Expected: `alembic upgrade head` succeeds; `alembic current` shows `0001 (head)`; the `\dt` listing includes
`alembic_version`. (Migrations target 5432 directly; that's intentional — the app still uses 6432.)
